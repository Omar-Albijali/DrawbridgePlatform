# Drawbridge Platform

منصة متعددة للمتاجر والتجارة الإلكترونية مبنية باستخدام Kotlin Multiplatform و Spring Boot و React.

## ما الذي سيتم تشغيله

- الواجهة الخلفية: `http://localhost:8080`
- تطبيق الويب: `http://localhost:3000`

يقوم تطبيق الويب بتمرير الطلبات الخاصة بـ `/api` و `/uploads` إلى الواجهة الخلفية، لذلك يجب تشغيل الخدمتين معًا أثناء تطوير الويب.

## مكونات المشروع

- `server`: الواجهة الخلفية المبنية بـ Spring Boot
- `webApp`: واجهة الويب المبنية بـ React و Vite
- `shared`: المنطق والنماذج المشتركة في Kotlin Multiplatform
- `composeApp`: واجهة Compose Multiplatform المشتركة
- `androidApp`: نقطة تشغيل تطبيق Android
- `iosApp`: نقطة تشغيل تطبيق iOS

## البدء السريع

افتح جذر المشروع في PowerShell ونفذ الخطوات بالترتيب.

### 1. استنساخ المستودع

```powershell
git clone <repository-url>
cd DrawbridgePlatform
```

إذا كان المشروع موجودًا لديك بالفعل، افتح المجلد مباشرة:

```powershell
cd <path-to-DrawbridgePlatform>
```

### 2. إنشاء ملفات البيئة المحلية

```powershell
./setup-env.cmd
```

هذا الأمر ينشئ الملفات التالية إذا لم تكن موجودة:

- `server/.env`
- `webApp/.env`

بعد ذلك افتح `server/.env` واضبط القيم المطلوبة التالية:

- `DB_USER`
- `DB_PASSWORD`
- `JWT_SECRET`

يمكن الإبقاء على باقي القيم الافتراضية للتطوير المحلي ما لم يطلب الفريق غير ذلك.

### 3. بناء الوحدة المشتركة

يجب بناء حزمة JavaScript الخاصة بالوحدة المشتركة قبل تشغيل تطبيق الويب لأول مرة:

```powershell
./gradlew.bat :shared:jsBrowserDevelopmentLibraryDistribution
```

### 4. تثبيت اعتمادات الواجهة الأمامية

```powershell
npm install
```

### 5. تشغيل الواجهة الخلفية

في نافذة PowerShell أولى:

```powershell
./gradlew.bat :server:bootRun
```

انتظر حتى يكتمل تشغيل الخادم، ثم تأكد أنه متاح على `http://localhost:8080`.

### 6. تشغيل تطبيق الويب

في نافذة PowerShell ثانية:

```powershell
npm run dev
```

ثم افتح `http://localhost:3000`.

## قائمة التحقق عند أول تشغيل

استخدم هذه القائمة إذا لم يعمل المشروع كما هو متوقع.

1. تأكد أن إصدار Java هو 21.
2. تأكد أن `server/.env` موجود ويحتوي على `DB_USER` و `DB_PASSWORD` و `JWT_SECRET`.
3. تأكد أن بناء الوحدة المشتركة تم بنجاح قبل أول تشغيل لتطبيق الويب.
4. تأكد أن `webApp/node_modules` موجود بعد تشغيل `npm install`.
5. تأكد أن الواجهة الخلفية تعمل على المنفذ `8080` قبل اختبار الصفحات التي تعتمد على الـ API.
6. تأكد أن تطبيق الويب يعمل على المنفذ `3000`.

## أوامر مفيدة

استخدم هذه الأوامر عند الحاجة. شغّل أوامر Gradle من جذر المشروع.

أعد بناء الوحدة المشتركة بعد أي تعديل في shared Kotlin:

```powershell
./gradlew.bat :shared:jsBrowserDevelopmentLibraryDistribution
```

بناء الواجهة الخلفية:

```powershell
./gradlew.bat :server:build
```

تشغيل اختبارات الواجهة الخلفية:

```powershell
./gradlew.bat :server:test
```

بناء تطبيق الويب:

```powershell
npm run build
```

## سير عمل Git

هذا المستودع يفضل استخدام rebase بدلًا من merge في العمل المعتاد على الفروع. الهدف هو الحفاظ على سجل خطي وأسهل في المراجعة.

### الفروع

أنشئ فرعًا قصير العمر لكل مهمة، ولا تعمل مباشرة على `master`.

### تحديث الفرع

عندما يتقدم `master`، حدّث فرعك باستخدام rebase. هذا يعيد تطبيق التزاماتك المحلية فوق أحدث سجل من المستودع البعيد ويتجنب إنشاء merge commits غير ضرورية.

الأسلوب الموصى به:

```powershell
git pull --rebase origin master
```

إذا توقف rebase بسبب تعارضات، قم بحل الملفات المتعارضة، ثم أضفها إلى stage وأكمل rebase. وإذا سارت العملية بشكل غير صحيح، ألغِ rebase للعودة إلى الحالة السابقة.

### دفع التغييرات

معظم عمليات الدفع يجب أن تكون عادية. بعد تنفيذ rebase، قد يرفض Git عملية الدفع لأن السجل المحلي تغير. في هذه الحالة، استخدم `--force-with-lease` بدلًا من `--force`.

- `git push --force-with-lease` هو الخيار الآمن لأنه يرفض الكتابة فوق تغييرات بعيدة لم تقم بجلبها بعد.
- `git push --force` أكثر خطورة لأنه قد يستبدل عمل شخص آخر على الفرع البعيد.

### القاعدة اليومية في هذا المستودع

- `rebase` هو أسلوب الدمج المفضل لفروع العمل.
- تجنب `merge commits` إلا إذا كانت هناك حاجة واضحة للإبقاء عليها.
- لا تقم بعمل rebase على فرع يعمل عليه أكثر من شخص بدون تنسيق مسبق.
